# SAGE-RL: A Multi-Agent Self-Evolving Framework for Enhancing LLM Reasoning

This project implements **SAGE-RL**, a multi-agent framework designed to enhance Large Language Model (LLM) reasoning capabilities through task-relative reinforcement learning. Inspired by advanced curriculum learning and policy optimization techniques, SAGE-RL orchestrates a system of specialized AI agents that collaboratively tackle complex problems, iteratively refining their strategies and improving problem-solving proficiency.

The core idea is to create a self-evolving system where LLMs learn from their own generated tasks and solutions, guided by a sophisticated reward mechanism. This involves:

1.  **Multi-Agent Orchestration:** Utilizing a framework like LangGraph to define a structured workflow between distinct LLM agents, each with a specific role (Challenger, Planner, Solver, Critic).
2.  **Curriculum Learning:** The "Challenger" agent dynamically generates increasingly difficult tasks, pushing the limits of the system's capabilities.
3.  **Chain-of-Thought (CoT) Reasoning:** The "Planner" agent is prompted to develop detailed strategies for solving tasks, fostering robust reasoning pathways.
4.  **Reinforcement Learning (REINFORCE++):** Rewards are calculated based on the quality of solutions and adherence to desired formats, guiding the LLM's learning process. This involves per-role normalization, KL-Divergence to prevent model drift, and careful policy updates.
5.  **Local Prototyping & Scaling:** The project is structured to allow for local development and testing on Apple Silicon (using MLX) and then scaled to high-performance computing (HPC) environments with CUDA (using vLLM and veRL) for distributed training and fine-tuning.
6.  **Continuous Improvement:** A logging and evaluation system tracks agent performance, task difficulty, and reward curves, enabling continuous monitoring and optimization of the learning loop.

By creating this multi-agent feedback loop, SAGE-RL aims to develop LLMs that are not only capable of solving challenging problems but also inherently understand how to learn and adapt to new complexities.

## Project Structure: The `agents` Folder

The `agents` directory is the heart of the SAGE-RL framework, housing the individual LLM agents and the orchestration logic that binds them together. Each file typically defines an agent's role and how it interacts with the shared `SAGEAgentState`.

### `agents/challenger.py`

This file defines the `challenger` agent. Its primary role is to generate new and challenging tasks based on a given set of reference examples. It uses a `challenger_policy` (a system prompt) to guide the LLM in creating these tasks, promoting a curriculum learning approach where task difficulty can evolve. The generated tasks are then added to the shared agent state for subsequent processing by other agents.

### `agents/critic.py`

The `critic` agent, defined here, evaluates the performance of the other agents—specifically the plans generated by the `planner` and the solutions provided by the `solver`. It assigns scores (1-10) for question quality, plan quality, and solution quality, along with rewards related to task difficulty and adherence to format. These scores and rewards are crucial for the reinforcement learning process, providing feedback that will eventually be used to update the underlying LLM.

### `agents/graph_workflow.py`

This file is responsible for orchestrating the entire SAGE-RL workflow using `langgraph`. It defines the state graph, connecting the various agents (`challenger`, `planner`, `solver`, `critic`, `reward`) in a sequential flow. It also initializes the shared backbone LLM and wraps each agent function with the model and a specific LoRA adapter name, enabling fine-tuning of individual agent behaviors. The `create_graph` function compiles this workflow, and it even includes functionality to visualize the graph.

### `agents/planner.py`

The `planner` agent is defined in this file. Given a set of tasks, its responsibility is to generate a concise, step-by-step plan for solving each task. It leverages a `planner_policy` to encourage chain-of-thought (CoT) reasoning, ensuring that the LLM develops a strategic approach before attempting to solve the problem. These plans are then stored in the shared agent state.

### `agents/reward.py`

This file contains the `reward_agent` function, which calculates the final rewards for each agent based on the scores assigned by the `critic` and predefined parameters (like `alpha`, `lambda_plan`, `w_p`, etc.). This module translates the qualitative scores into quantitative rewards that will drive the reinforcement learning process, guiding the Challenger, Planner, and Solver to improve their respective functions.

### `agents/solver.py`

The `solver` agent is implemented here. Its function is to take a given task and its corresponding plan (generated by the `planner`) and produce a detailed solution. It uses a `solver_policy` to guide the solution generation, aiming for accurate and comprehensive answers. The generated solutions are then added to the shared agent state, ready for evaluation by the `critic`.

### `agents/tools.py`

This file is intended to house various utility functions or "tools" that might be used within the SAGE-RL framework, particularly for reward mechanisms. Currently, it defines placeholder tool functions like `reward_challenger`, `reward_planner`, and `reward_solver`, which illustrate how external tools could be integrated to help construct messages or potentially contribute to more complex reward calculations. At this stage, they primarily demonstrate the structure for defining such tools.


## Project Walkthrough: `main.py`

The `main.py` file is the central script that initializes the SAGE-RL framework, sets up the multi-agent workflow, and kicks off the iterative learning process. It brings together all the components defined in the `agents` folder and the configuration settings.

Here's a step-by-step breakdown of its functionality:

1.  **Environment Setup:**
    *   `load_dotenv()`: This line loads environment variables from a `.env` file. This is crucial for managing sensitive information or configuration parameters (like API keys, data directories, model names) without hardcoding them directly into the script.
    *   `data_dir = os.getenv("DATA_DIR")` and `greeting = os.getenv("GREETING")`: These retrieve specific environment variables, likely for data paths and an initial greeting for the model.

2.  **Database and Logging Initialization:**
    *   `clear_db()`: Clears any existing database entries, ensuring a fresh start for each run. This is important for reproducible experiments and preventing stale data from affecting new learning sessions.
    *   `init_database()`: Initializes the project's database, which is used for persistent storage of agent states, rewards, and other relevant data throughout the learning process.
    *   `logger.info(...)`: Various logging statements are used throughout `main.py` to provide real-time feedback on the execution flow, making it easier to monitor the system's progress and debug any issues.

3.  **Dataset Loading:**
    *   `train_data = get_data(split="train")`: The script loads the GSM8K dataset (a dataset of grade school math problems) for the "train" split. This dataset serves as the initial source of tasks and examples that the Challenger agent will use to learn and generate new tasks, and against which the Solver's performance can be benchmarked.

4.  **Backbone Model Initialization:**
    *   `model = get_backbone()`: This is where the core Large Language Model (LLM) for the project is loaded. Based on the logs, it's configured to use `llama3:8b` (or `TinyLlama:1.1b` for testing, as indicated in the logs). This `model` instance acts as the shared "brain" for all the agents, with LoRA adapters likely being used to specialize its behavior for each agent's role.
    *   `response = model.test_model(greeting)`: A quick test is performed on the loaded backbone model to ensure it's functioning correctly by invoking it with a simple `greeting`.

5.  **Agent Workflow Creation:**
    *   `graph = create_graph()`: This crucial step calls the `create_graph` function from `agents.graph_workflow`. As described earlier, this function constructs the multi-agent interaction graph using LangGraph, defining the sequence and dependencies between the Challenger, Planner, Solver, Critic, and Reward agents.
    *   `logger.info(...)`: Confirms that the graph has been created and that a visualization (`agent_workflow.png`) has been saved, which is incredibly helpful for understanding the agent flow.

6.  **Initial Training Loop / Simulation:**
    *   `query = [...]`: Defines a list of initial seed tasks or example queries that the Challenger agent will use as inspiration.
    *   `messages = [HumanMessage(content=f"Generate 3 different mathematical tasks similar to the {query}")]`: Prepares the initial message for the Challenger agent, instructing it to generate new mathematical tasks similar to the provided examples.
    *   `initial_state = SAGEAgentState(...)`: An `SAGEAgentState` object is created. This state object is central to the LangGraph workflow, holding all the dynamic information—messages, input, and the list of tasks—that gets passed and modified by each agent as the workflow progresses.
    *   `response = graph.invoke(initial_state)`: This is where the magic happens! The `invoke` method triggers the execution of the entire multi-agent workflow defined in `graph_workflow.py`. The `initial_state` is fed to the first agent (Challenger), which then passes its output to the next agent, and so on, until the `END` node is reached.

7.  **Output and Reward Display:**
    *   The final part of `main.py` checks the `response` from the graph invocation. If new tasks were generated and processed, it iterates through them and prints the `rewards` associated with each task. This provides an immediate glimpse into the outcome of the multi-agent evaluation cycle for the generated tasks. If no tasks are found, it indicates that.

8.  **Entry Point Execution:**
    *   `if __name__ == "__main__": main()`: This standard Python construct ensures that the `main()` function is called only when the script is executed directly, not when it's imported as a module into another script.

In essence, `main.py` orchestrates a single pass through the SAGE-RL feedback loop, from task generation to solution and reward calculation, setting the stage for more advanced reinforcement learning training to take place in subsequent iterations or with specialized training scripts.
